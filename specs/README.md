# Specs

Contract-first. The YAML files are the source of truth. Implementation must conform to the
spec — not the other way around.

---

## Services

| Spec | Service | Image under test |
|------|---------|-----------------|
| `public-api.yaml` | `WeatherPlaylist.Api` (.NET) | `cgr.dev/chainguard/dotnet-runtime` |
| `playlist-engine.yaml` | `playlist-engine` (Go) | `cgr.dev/chainguard/go` |
| `mock-services.md` | `mock-spotify` + `mock-owm` (Go) | n/a — local HTTPS mocks, not under CVE test |

### Call flow

```
Browser
  └─ POST /playlist/generate ──► WeatherPlaylist.Api (.NET)
                                      ├─ GET OpenWeatherMap API  (outbound HTTP — .NET CVE surface)
                                      └─ POST /playlist/generate ──► playlist-engine (Go)
                                                                          └─ Spotify API  (outbound HTTP — Go CVE surface)
```

The .NET service owns the public contract and all UI concerns.
The Go service owns all Spotify interactions and the weather→mood mapping.

## Header Conventions

These headers flow through every request and response in the system.

### X-Correlation-Id
- **Origin:** Generated by the .NET service on receipt of every inbound request (UUID v4).
  If the caller supplies one, it is used as-is.
- **Flow:** .NET → Go request header → Go echoes on response → .NET echoes on response.
- **On errors:** Included in `ProblemDetails.correlation_id` body field and response header
  so errors can be traced across both services in logs.

### X-Experiment-Id
- **Origin:** Set by the k6 stress test script (or the caller) to reference a manifest
  in `specs/experiments/`.
- **Flow:** Passed through request body (`experiment_id`) and header. Echoed back on all
  responses in header and body.
- **Purpose:** Links every request in a stress test run to the specific CVE experiment
  being executed. Enables post-run filtering of logs and k6 results by experiment.

### X-Api-Source
- **Origin:** Set by the .NET service only — never by the Go service.
- **Value:** `GO-API-PASSTHROUGH`
- **When:** Present on any 4xx/5xx response that the .NET service is passing through
  directly from the playlist-engine (Go). Absent on errors originating in .NET itself.
- **Purpose:** Lets consumers and monitoring distinguish Go-origin failures from
  .NET-origin failures without parsing the body.

### X-ARBITRARY-MOCK

This header has two distinct roles depending on direction.

#### On requests (caller → .NET → Go): mock control

| Value | Effect |
|-------|--------|
| `spotify` | Go service routes Spotify API calls for this request to `SPOTIFY_MOCK_BASE_URL`. The HTTPS chain is preserved — TLS CVE surface still fires. Has no effect if `SPOTIFY_MOCK_BASE_URL` is not set (safety gate). |

Set by k6 or any client. The .NET service passes it through to the Go service unchanged.

#### On responses (service → caller): mock status

| Value | Meaning |
|-------|---------|
| `weather` | OWM was mocked (active for all responses when `OPENWEATHERMAP_API_KEY` is absent at startup) |
| `spotify` | Spotify was mocked for this request |
| `weather,spotify` | Both were mocked |

Set by the .NET service to aggregate the mock state for the full request chain.

**CVE validity rule:** Any response that carries `X-ARBITRARY-MOCK` on the response
(in any value) indicates a dependency was short-circuited without a live TLS connection.
k6 must assert this header is absent for any result to count as a valid CVE surface
measurement — unless using dedicated external mock servers (see `specs/mock-services.md`),
which preserve the HTTPS chain.

## Stress Test Targets

Each endpoint exercises a distinct slice of the CVE surface. k6 scenarios target them
individually and in combination.

| Endpoint | Service | CVE surface exercised | Scenario use |
|----------|---------|----------------------|-------------|
| `POST /playlist/generate` | .NET | openssl (OWM TLS) + glibc + Go surface transitively | **Primary** — full end-to-end |
| `GET /weather` | .NET | openssl (OWM TLS) + glibc | Isolates .NET → OWM leg |
| `POST /recipe` | Go (direct) | glibc only (no Spotify calls) | Isolates Go compute, no openssl, no rate-limit noise |
| `POST /playlist/generate` | Go (direct) | openssl (Spotify TLS) + glibc | Isolates Go → Spotify leg |
| `GET /health` | both | Minimal | Warmup / liveness baseline |

## Auth

**Spotify:** Pre-authenticated refresh token in `SPOTIFY_REFRESH_TOKEN` env var.
The Go service handles token refresh internally. Callers never supply tokens.
Rationale: eliminates OAuth redirect complexity; keeps the HTTP client path — the primary
CVE surface — as the only variable under stress.

**OpenWeatherMap:** API key in `OPENWEATHERMAP_API_KEY` env var, sent as a query param.
If the key is absent, the service falls back to `MockWeatherService` — a static set of 7
conditions selected deterministically by lat/lon + UTC hour. All responses in mock mode
carry `X-ARBITRARY-MOCK: weather`.

## Service Discovery

The .NET service resolves the Go service via Aspire's built-in service discovery using
the resource name `playlist-engine`. No hardcoded ports or IPs in application code.

---

## Experiment Tracking

Each stress test run is tied to an experiment manifest in `specs/experiments/`.

The manifest defines:
- Which CVEs are under test
- Which image variant (Chainguard vs standard)
- Target endpoints and load profile
- Success criteria

The `experiment_id` field in every request body and `X-Experiment-Id` header provides
the link from a live request back to its manifest. This means every request, response,
and error in the logs is traceable to a specific CVE experiment.

See [`experiments/`](experiments/) for the manifest format and index.

## Decisions Log

| # | Decision | Rationale |
|---|----------|-----------|
| Q1 | `GET /playlist/{playlistId}` removed from v0.1 | Adds state management irrelevant to CVE comparison; can be re-added after baseline is established |
| Q2 | Error passthrough with `X-Api-Source: GO-API-PASSTHROUGH` + `X-Correlation-Id` | Preserves signal for k6 (429 stays 429); correlation ID enables cross-service log tracing |
| Q3 | Always create new playlist — no caching | Caching would suppress Spotify HTTP calls under load, defeating the CVE surface exercise |
| Q4 | UI endpoint not in spec | Static HTML is an implementation detail; `GET /` does not belong in a machine-readable API contract |
| Q5 | Mock weather fallback + `X-ARBITRARY-MOCK` header | OWM registration blocked on DNS; mock keeps the system runnable for development and smoke testing without real credentials. Header makes mock mode unambiguous to all consumers. |
| Q6 | Dedicated mock servers (`mock-spotify`, `mock-owm`) + per-request `X-ARBITRARY-MOCK: spotify` header routing | Spotify rate limits are the dominant bottleneck for high-VU stress runs. In-process mocks bypass TLS entirely, invalidating CVE measurements. Dedicated HTTPS mock servers with a local CA preserve the full TLS handshake while removing external rate limits and credential requirements. See `specs/mock-services.md`. |
